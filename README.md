# 卒業制作 README

---

## 1. サービス概要

婚活・恋愛中の社会人が、出会った人への感想・気づきを記録することで、自分の恋愛における価値観と優先順位を可視化できるアプリ。「次の人と会う前に、自分が何を大切にしているか」を整理し、相手ごとのマッチ度を項目別スコアやランキングで確認できる。

| 項目 | 内容 |
|------|------|
| **誰の課題を解決するか** | 婚活中の20代後半〜30代前半の社会人 |
| **どんな課題か** | 人に会うたびに感じることはあるが、自分の優先順位が言語化できていない |
| **どう解決するか** | 出会いの記録を積み重ねることで、自分の価値観を浮かび上がらせる |

---

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

- **いつ：** 婚活を始めて、複数の人と会い始めた時期
- **どんな状況で：** いろんな人に会う中で、「この人は何か違う」「あの部分は嫌だったな」という感覚が生まれた。その都度メモを取っていた
- **どんな気持ちになったか：** メモはあるのに、結局「自分が何を大事にしているのか」がまとまらなかった。次の人に会うときも判断軸が曖昧なまま会ってしまっていた

### 2-2. なぜそれが「気になった」のか

人に会うたびに感情は動くのに、それが次の判断に活かせていないことへの違和感があった。「好き・嫌い」という感覚は確かにあるのに、それを自分の言葉で整理できないもどかしさ。自分の中の優先順位を自分で考えることの難しさを感じた。

---

## 3. 課題の整理

### 3-1. 表に見えている困りごと

出会った人に対して感じたことをメモしているが、バラバラで整理できていない。

### 3-2. 本当に解決したい課題は何か

- **なぜ？** → メモがバラバラだと、次に会う人への判断軸が作れないから
- **なぜ？** → 判断軸がないと、「この人でいいのかな」という確信が持てないから
- **なぜ？** → 確信が持てないと、大事な決断（交際・婚約など）に踏み出せないから

> **本当に解決したい課題：自分の恋愛・婚活における価値観と優先順位を言語化・整理できていないこと**

---

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

**年齢設定**
- **年齢：** 20代後半〜30代前半
- **理由：** 結婚を意識して婚活を始めるが、まだ「自分が何を求めているか」が言語化できていない年代。社会人として多忙な中での婚活なので、感覚的に動きがちで記録・整理の習慣がない

**生活・状況設定**
- **状況：** 結婚相談所・マッチングアプリ・紹介などで複数人と同時進行で会っている
- **理由：** 複数の人を比較する場面があるからこそ「誰が自分に合うか」の判断が難しくなる。このアプリが最も役に立つ状況

**環境設定**
- **環境：** スマートフォンでいつでも手軽に記録できる
- **理由：** デートの帰り道など、感情が新鮮なうちにすぐ記録することが重要。PCでの操作は習慣化しにくい

### 4-2. 自分とユーザーの距離

**自分自身**

自分がまさに体験した課題から生まれたアイデアのため、ユーザーの感情や状況を解像度高くイメージできている。

### 4-3. 実際に使われる可能性について

- **最初に使ってくれそうな人：** 婚活中の同世代の友人・知人
- **どんな場面でこのアプリを思い出すか：** デートの帰り道に「今日どうだったっけ」と振り返るとき
- **今代わりに使っているもの：** メモアプリ・LINE自分宛て・手帳
- **置き換える理由：** 単なるメモと違い、記録が蓄積されると自分の価値観が可視化される。人物ごとのスコアや自由タグによる気づきが、既存ツールにはない差別化ポイントになる

### 4-4. 導入・継続の流れ

- **きっかけ：** 「婚活 整理 アプリ」などで検索、または婚活仲間からの口コミ
- **最初の一歩：** アカウント登録 → 価値観の初期項目を設定（デフォルト項目あり）→ 最初の1人を記録する
- **使い続ける理由：**
  - **婚活がうまくいったユーザー：** このアプリのゴールは交際開始ではなく結婚の決断まで。婚活とは結婚するための活動であり、交際中も過去の婚活記録と照らし合わせながら「この人でいいのか」を確認し続けられる
  - **なかなか出会えないユーザー：** 記録が増えるほど「自分が何を大切にしているか」が言語化されていく。次の出会いへの判断軸が育っていく感覚が、続けるモチベーションになる

> このアプリの入口は「婚活の記録」だが、出口は「自分を知ること」。婚活が終わっても価値が残る設計を目指す。

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

- **サービス名：** Koilog
- **URL：** https://apps.apple.com/jp/app/koilog-%E6%81%8B%E3%83%AD%E3%82%B0-%E6%81%8B%E6%B4%BB-%E5%A9%9A%E6%B4%BB%E3%81%AE%E3%81%9F%E3%82%81%E3%81%AE%E8%A8%98%E9%8C%B2%E3%82%A2%E3%83%97%E3%83%AA/id6755999749
- **どんなことができるか：**
  - 相手情報をカード形式で整理。基本情報・ステータス・メモを記録可能
  - 日時・場所・金額を記録するデート＆費用ログ機能
  - 写真や気持ちを日記として保存できる思い出アルバム機能
  - Face ID／Touch ID／パスコードによるプライバシー保護
  - データは端末内のみ保存する完全オフライン設計

### 5-2. それでも自分が作りたい理由

Koilogは記録・管理ツールとして優れているが、「記録をもとに自分の価値観を整理する」という機能がない。デートのログや費用は残せても、「自分が何を大切にしているか」という気づきには繋がらなかった。記録が増えても「で、自分は何を重視しているんだっけ？」という問いには答えてくれない点に不満を感じた。

### 5-3. 差別化を一文で決める

> 「婚活中の社会人が、出会いを重ねるほど自分の価値観が言語化されていく」瞬間を一番助けるアプリ。スコアによる定量評価だけでなく、自由タグによる感情の言語化を組み合わせることで、「なんとなく嫌」を言葉にする体験を提供する。

## 6. このサービスで提供したい価値

### 6-1. ユーザーの変化

- **行動の変化：** デートの後に感想を記録する習慣ができる
- **気持ちの変化：** 「なんとなく合わない」が「〇〇という価値観が合わない」と言語化できるようになる
- **考え方の変化：** 「相手に求めること」ではなく「自分が大切にしたいこと」を軸に婚活できるようになる

### 6-2. 価値を一文で表す

> 「このサービスは、婚活中の社会人が、出会いの記録を通じて自分の価値観を発見し、自信を持って決断できるようになるサービス」

## 7. このアプリで実現すること

このアプリの核は「記録する」行為にあり、他の機能はすべて、記録の蓄積があってはじめて価値を持つ

### MVPで作る機能

- ユーザー登録・ログイン
- 会った人の記録（名前・日付・良かった点・気になった点を自由記述）
- 価値観項目の設定（デフォルト項目あり・後から編集可能）
- 各項目の5段階スコアリング入力
- 自由タグの作成・記録への付与
- プリセットタグの用意（10個程度・ネガティブ7：ポジティブ3のバランスで選定。自分の体験・Yahoo知恵袋の婚活相談・AIを活用してリサーチし決定する）

### 本リリースで作る機能

- 同じタグが複数人に重複したときの通知（例：「このタグが2回登場しています。あなたはこの点を大切にしているのかもしれません」）
- 1人目の記録直後のフィードバック表示（「あなたが最も気になったのは〇〇でした。次の人と会うときはこの点を意識してみましょう」）
- 相手ごとのマッチ度スコア自動計算・表示
- スコアの単純比較表示（例：「前の人より誠実さのスコアが高いです」）
- 価値観マップのレーダーチャート表示
- 項目別ランキング（例：「誠実さ」部門1位はAさん）
- 価値観サマリーの自動生成（例：「あなたが最も重視しているのは〇〇です」）
- 感想の殴り書きからAIがタグを自動提案する機能

## 8. このアプリの懸念点とその対策

### 懸念点①：プライバシーへの不安からユーザーが記録をためらう可能性

- **何が問題になりそうか：** 婚活相手への感想・スコア・気になった点など、記録の内容そのものを外部サービスに登録することへの抵抗感
- **なぜそう思うか：** ユーザーが本当に不安に思うのは名前よりも「書いている内容」であり、「データが漏れたら」という不安から使うのをためらう可能性がある。競合のKoilogが「完全オフライン」を売りにしていることからも、このニーズの大きさがわかる
- **対策：**
  - 名前は仮名・ニックネームで登録できるようにする
  - ログイン必須・他ユーザーからの完全非公開設計を実装し、外部共有機能は持たない
  - トップページに「データは暗号化されています」「第三者に共有されません」「いつでもデータを削除できます」を明示する
  - その上でサーバー保存のメリット（機種変更時のデータ保持・マルチデバイス対応）も伝え、不安をメリットに転換する

### 懸念点②：記録が続かず、データが少ないと価値を実感できない

- **何が問題になりそうか：** 数人しか記録していない状態では「ランキング」も「価値観マップ」も機能しない
- **なぜそう思うか：** 価値が実感できるのはある程度記録が溜まってからで、初期離脱が起きやすい
- **対策：**
  - ポジティブな価値観の発見よりも、ネガティブな条件の言語化の方が少ない記録数でも実感しやすいという特性を活かす
  - デートの帰り道に記録したくなるのは主に「気になった点」であることを踏まえ、プリセットタグはネガティブ多めで用意する
  - プリセットタグと自由タグを組み合わせることで、言語化のハードルを下げる

## 9. 今後の展開・発展の方向性

### 今の段階で決まっていること

- 記録が一定数を超えたら「あなたの価値観ランキングTop3」を自動生成する機能

- **交際中モード：** 婚活のゴールは結婚の決断までのため、交際中モードは婚活の延長として位置づける。婚活中（複数人比較モード）と交際中（1人との深掘りモード）をUIで切り替える設計を検討する
- **プリセットタグの内容：** 実装後はユーザーが作った自由タグを集計してプリセットに随時追加していく
- **価値観の変化の可視化：** スコアの平均値比較やタグの出現頻度比較など、実装方法はMVP後に検討する

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

| 項目 | 内容 |
|------|------|
| **フレームワーク** | Ruby on Rails |
| **DB** | PostgreSQL |
| **デプロイ先** | Render または Fly.io |
| **使用予定ライブラリ** | Chartkick（グラフ表示）、Devise（認証） |

### 10-2. 技術選定の理由

- RUNTEQカリキュラムで学んだRailsをベースに実装することで、キャッチアップコストを最小化する
- **チャレンジしたい点：** レーダーチャートやランキング表示など、データの可視化機能
- **不安な点：** スコアリングのロジック設計と、直感的なUI/UXの両立。またChartkickはレーダーチャートに非対応のため、本リリースでのレーダーチャート実装にはChart.jsの直接利用またはrails_charts gemの導入を検討する必要がある

### 10-3. キャッチアップ不足の懸念

MVPはRailsのみで構築し、グラフ表示はChartkickなど既存のgemを活用して実装コストを下げる。ReactやNext.jsへの移行は本リリース後に検討する。
